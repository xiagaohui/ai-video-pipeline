# 视频制作经验沉淀

## 2026-06-22 多 Agent 第 1 集复盘

这次最大的问题不是渲染，而是开头和配图没有站在观众视角。

原先用“登录”“接口改造”做开头，画面里出现参数兼容、调用方之类的词。它们对工程师并不陌生，但和视频立意“多 Agent 要不要上”不是同一个问题。观众刚点进来时，还没进入具体案例，先看到过窄的实现细节，会误以为视频在讲接口设计。

修正后的规则：

1. 封面、标题和第一幕必须直接回答同一个观众问题。
   这集的问题是：`多 Agent 要不要上？`
2. 具体案例只能作为解释载体，不能抢走主问题。
   可以写 `一个功能需求`，不要一上来写 `接口改造`、`登录系统`、`参数兼容`。
3. 每一幕之间要有显性连接。
   例如：先问要不要上多 Agent -> 再说不要默认上 -> 再解释为什么会出问题 -> 最后给 4 个判断信号。
4. 抽帧图必须能独立讲清楚逻辑。
   不听音频，只看图，也要知道“这张图为什么接在上一张后面”。
5. 用户说“奇怪”“看不懂”“这是什么”时，不只修当前画面。
   要把问题翻译成规则，写回规范，再检查整条视频有没有同类问题。

## 封面经验

封面不是资料目录。封面要回答“用户为什么要点开”。

合格封面至少满足：

- 大字能一眼看懂，不靠小字解释主题。
- 有钩子：纠偏、反差、痛点、方法、结果，至少命中一种。
- 能看出和视频立意的关系。
- 图内元素不重叠、不贴边、不被平台裁切。
- 信息图封面也必须有阅读顺序，不能只堆模块。
- 竖屏封面不能只做半屏信息图。为了避开平台遮挡而留安全区是对的，但不能把安全区变成大片空白；底部必须放观众收益、结论或行动建议。
- 封面标题不能写给作者自己看。像“任务形状”“协作模式”“万能钥匙”这类词，作者知道什么意思，观众未必知道。封面大字优先写观众已经有感觉的问题：越做越乱、上线翻车、乱改代码、调研废话。
- 当前封面标题统一采用“权威来源给出的具体方法/案例 + 这个方法解决的痛点”。大厂名不能当装饰，必须说明它提供了什么可用东西：流程、模式、官方方法、权限设计、实战案例或架构。
- 更好的标题要有反差：权威团队把同类任务跑稳了，但观众自己的 AI / Agent 正在翻车。封面大字优先采用“大厂做成了什么 + 你这里为什么翻车 + 缺什么”的结构，例如“Anthropic 让 16 个 Claude 写出 10 万行代码；你的 AI 写乱，是因为缺测试和隔离”。
- 反差必须靠同一个原因连起来。不要写成两个松散句子：`Claude 给了模式，为什么你乱？`；要写成因果句：`Claude 先分模式，所以能控住协作；你越用越乱，是因为没先选模式`。
- “痛点问题”要站在观众身上，而不是站在案例旁边。像“为什么没有翻车”还是案例视角；改成“多 Agent 写代码总改乱？”才是观众视角。

不合格表现：

- 只写资料名、功能名、公司名。
- 底部标签全是内部章节名。
- 画面很专业，但普通观众不知道它在解决什么问题。
- 封面承诺和视频前 15 秒讲的不是同一件事。

## 口播去 AI 味经验

短视频口播要像一个懂行的人在解释，不像模型在输出结构化答案。

默认处理顺序：

1. 先保证立意和逻辑线对观众有价值。
2. 再用 `writing-humanizer` 清理模板句、讲义腔和过整齐段落。
3. 如果是用户已经确认过的稿子，用 `de-ai-prompt-enhancer` 做保真修补。
4. 最后用 `humanizer-zh` 或 `stop-slop` 检查是否还有宣传腔、空话和提纲腔。

口播不能为了“像人”而乱加情绪、段子或个人经历。自然感来自具体、清楚和节奏变化，不来自装作随口一说。

## 画面经验

配图的第一目标是让观众看懂。

- 每张图只讲一个核心意思。
- 图内要有阅读顺序：先看哪、再看哪、最后带走什么。
- 前后图要沿同一条逻辑推进，不能每张图都像独立 PPT。
- 复杂概念要落到现实名词：需求、代码、测试、验收、日志、权限、成本、时间。
- 如果画面需要口播解释很多，说明图本身没做好。

## 2026-06-23 参考过程图复盘：清晰又有冲击力的画面长什么样

用户补充了一组 `Claude Code + Codex` 视频过程图。它们的价值不只是“深色科技风”，而是提供了一套更适合知识类短视频的信息表达方法。

可复用规则：

1. **每页只承担一种讲解功能。**
   有的页负责对比两个角色，有的页负责解释什么时候用某个工具，有的页负责给流程，有的页负责列风险检查项。不要每页都做同款三卡片。
2. **大标题必须是观众能直接理解的判断。**
   例如 `把两个 Agent 放在不同位置上`、`适合当主力干活`、`先读项目结构，再给实施方案`。这些句子都是动作和建议，不是抽象概念。
3. **副标题只补一句解释，不抢主标题。**
   主标题负责停留，副标题负责降低理解成本。副标题不要再塞第二个观点。
4. **当前重点要用发光边框或高亮态明确标出来。**
   列表、流程、对比卡里必须让观众知道“现在讲到哪一步”。如果所有元素一样亮，观众会迷路。
5. **底部字幕区不承担核心信息。**
   平台控件和字幕容易挡住底部。底部可以放强化句，但关键判断必须在画面主体中已经讲清楚。
6. **颜色承担语义。**
   蓝色偏主流程/主力工具，紫色偏审查/第二视角，橙色偏风险/警示，绿色偏通过/推荐。不要随意换色。
7. **图标必须服务现实动作。**
   终端代表执行，放大镜代表审查，盾牌代表风险，文件代表改动记录，流程节点代表步骤。图标不是装饰。
8. **横屏参考转竖屏时，要保留结构，不照搬比例。**
   横屏里的“标题在左上，主体横向展开”转成竖屏时，可以变成“上方大标题，中间主视觉，下方流程/卡片”，但仍要保持同样的阅读顺序。

后续多 Agent 系列的画面优先套用这些页面类型：

- **对比页**：两个 Agent / 两种做法 / 错误做法与正确做法。
- **步骤页**：1-4 个动作，用高亮框显示当前步骤。
- **流程页**：从做方案到实现、审查、复审的路径，节点数量控制在 3-5 个。
- **风险清单页**：用感叹号、盾牌、勾叉明确哪些问题需要被查。
- **例子页**：先给真实任务，再展示工具如何介入，不要只讲抽象原则。

这组图对后续视频的提醒：专业感来自“信息组织清楚”，不是来自堆光效。视觉冲击力应该帮观众更快看懂，而不是把内容变复杂。

## 2026-06-24 多 Agent 第 3 集复盘：模式图不能做成文字卡片

这次 EP3 讲 Claude 官方的 5 种 multi-agent coordination patterns。初版的问题是：每个模式页虽然写了适用场景、优势、风险，但画面主要是文字卡片。观众看完仍然不知道“这个模式到底长什么样”，更难比较五种模式之间的差异。

用户反馈的核心不是“少写几个字”，而是：画面应该先展示模式架构，再解释优劣势和场景。模式类内容如果只列定义，会变成 PPT 摘要；只有画出结构，观众才知道它怎么运转。

修正后的标准：

1. **模式页先画架构，不先摆文字。**  
   主视觉必须表达关系：谁触发、谁处理、谁验证、谁汇总、失败怎么回流。
2. **每种模式必须有不同图形关系。**  
   生成验证是 `生成 -> 验证 -> 通过 / 重写`；子 Agent 是 `主 Agent -> 多个专项 Agent -> 统一结论`；Teams 是多条任务线并行；总线是 `事件 -> Bus -> 对应 Agent`；共享状态是多个 Agent 读写同一状态。
3. **文字只做决策辅助。**  
   每页保留三枚短标签：`适用 / 优势 / 风险`。每枚只写一个短句，不能变成段落。
4. **必须给现实场景。**  
   观众需要知道模式用在哪，不要只记英文名。比如 `代码生成 + 测试验收`、`PR review：安全 / 测试 / 架构分查`、`告警 / 工单 / 日志自动分发`。
5. **抽帧总览必须能分辨五种模式。**  
   不听音频，只看 8 张抽帧，也要知道每个模式和上一个模式有什么区别。

本次沉淀出的通用规则：

- 架构类、模式类、方法论类视频，过程图默认采用 `架构图 + 适用 + 优势 + 风险 + 场景`。
- 如果画面里大部分信息都是解释文字，而不是结构关系，默认需要重画。
- 不要用同一套卡片模板给每个章节硬换文字。章节之间的图形关系要跟内容逻辑一起变化。
- 底部口播字幕不是画面解释的替代品。字幕可以帮助听懂，但不能承担“图为什么成立”的主要解释。

## 2026-06-24 发布文案复盘：短标题要真的短

视频发布时，用户要求短标题不能超过 16 个字符，并指出 `Claude官方5种Agent模式` 仍然偏长。这里的经验是：短标题不是封面标题，也不是完整选题句。它只负责在平台列表里给一个可搜索、可识别的入口。

后续规则：

- 视频号短标题默认 12 个汉字以内；如果用户说 16 个字符以内，英文、数字、空格和标点都算字符。
- 短标题优先保留对象和问题，删除来源、修饰和解释。
- 对于这类视频，更合格的短标题方向是：`Agent模式怎么选`、`5种Agent模式`、`多Agent怎么选`。
- 长标题、封面大字、视频简介可以承载完整钩子；短标题不要硬塞全部信息。

## 2026-06-24 多 Agent 系列复盘：先分清核心文章和扩展选题

这轮多 Agent 系列暴露出一个流程问题：同一个主题会同时存在“核心文章来源视频”和“扩展学习顺序视频”。如果不在一开始分清，后面就会出现“到底还有哪个视频没做”的混乱。

后续规则：

1. **先建立来源清单，再建立视频清单。**  
   每条视频必须绑定一个主来源，记录 URL、标题和视频立意。一个视频可以参考多篇文章，但必须有一个主来源，否则封面和逻辑线容易发散。
2. **核心系列和扩展系列分开管理。**  
   例如本轮核心 5 篇是 Cognition、Claude 两篇、Google Codelab、Anthropic compiler；Claude Code Subagents、Subagent 权限、Anthropic Research 系统属于扩展选题，不应混在同一个完成状态里。
3. **正式交付目录才算完成。**  
   只有同时包含 `final/`、`cover/`、`publish/videohao/`、`qa/`、`process/` 或 README 的独立文件夹，才算可发布完成。Remotion `out/` 里的临时视频不算正式交付。
4. **视频编号不能覆盖内容判断。**  
   文件夹编号和系列编号可以后期整理，但观众只关心这一条解决什么问题。最终发布信息必须优先写主题、立意、观众收益和来源。
5. **做完一条就更新系列总表。**  
   每条视频结束后，立即把状态、主来源、成片路径、封面路径、发布标题和 QA 路径写回系列表，避免靠记忆追踪。

对观众来说，系列视频的价值不是“看完一堆资料”，而是沿着一个清晰学习路径解决问题：先判断要不要上多 Agent，再知道怎么选模式，再学会生产化，最后理解大型工程案例背后的测试、隔离和接管机制。

## 2026-06-24 Claude Tag 复盘：新功能视频要翻译成落地判断

这条视频的主来源是 Anthropic 的 Claude Tag 发布文和 Agent identity 解读。初看像产品新闻：Claude 可以进 Slack，被团队 `@Claude` 调用。但如果只讲“发布了什么”，观众最多知道一个新功能，无法判断这件事和自己有什么关系。

这类新功能视频的正确做法，是把新闻事实翻译成观众能使用的判断框架。本条最终立意从“Claude Tag 发布”改成“团队 AI 权限不能再按个人账号管”，核心价值就变清楚了：以后把 AI 接进 Slack、飞书、企微、代码库或知识库时，要先想清楚身份、空间、工具和审计。

后续规则：

1. **新功能不是立意，功能背后的变化才是立意。**  
   `Claude Tag 进 Slack` 是事实；`团队 AI 需要 Agent identity` 才是观点；`上线前问四个问题` 才是观众收益。
2. **封面要有“旧模型 vs 新模型”的反差。**  
   本条封面用 `借某个人账号权限 ≠ 给 Agent 独立身份`，比单写 `Claude Tag 发布` 更有传播性，也更容易让人明白为什么要看。
3. **官方数字必须加边界。**  
   `65%` 只能表达为 Anthropic 内部版本和产品团队的结果，不能说成所有团队使用 Claude Tag 都能达到。
4. **协作工具名要说准确。**  
   当前可用范围只说 Slack beta、Claude Enterprise 和 Team 用户；不能暗示已经支持飞书、企微或所有企业用户。
5. **权限类内容要落到现实对象。**  
   观众能懂的是频道、DM、代码库、日志、客户资料、审批材料、预算上限；不要只停留在“身份模型、访问控制、治理体系”。
6. **结尾必须给检查清单。**  
   新功能解读的收束最好不是“值得关注”，而是“你可以拿这几个问题去检查自己的团队”。本条沉淀为四问：用谁的身份、在哪个空间、能碰什么、谁能停掉。

## 2026-06-25 Claude Tag 二次复盘：机制不是立意，角色变化才是立意

这次用户进一步指出：Claude Tag 的理念透传的是 `AI 员工`，不只是做了一个 Agent。这个反馈很关键，因为上一版虽然已经从“功能发布”上提到了“权限治理”，但仍然容易让视频停在 `Agent identity` 这个机制层。机制是支撑，不能替代立意。

最终封面改为：

> Claude Tag核心理念：AI从个人助手变成组织员工

这个标题比“Claude Tag 透传 AI 员工入职”更清楚。它不是在解释作者脑子里的概念，而是直接告诉观众发生了什么角色变化：AI 不再只是个人聊天框里的助手，而是进入组织工作环境，开始有岗位、工位、工牌、工具、记录和接管规则。

本次新增规则：

1. **机制不是立意。**  
   `Agent identity` 是工牌机制，`Slack @Claude` 是入口功能，`AI 从个人助手变成组织员工` 才是对观众有价值的主判断。后续所有新功能视频都要先问：这个功能背后代表哪种工作方式变化？
2. **理念型封面优先写“从 A 到 B”。**  
   当内容讲设计理解，而不是教程或评测时，封面可以采用 `产品名核心理念：旧角色/旧方式 -> 新角色/新方式`。例如本条是 `AI从个人助手变成组织员工`。这种句式比“透传”“底层逻辑”“本质”更容易被普通观众转述。
3. **AI 员工类视频用六问落地。**  
   不能只说“AI 员工来了”。必须给观众一套检查清单：岗位、工位、工牌、工具、记录、接管。审计、日志和责任边界可以放在“记录”和“接管”里讲清楚。
4. **真实组织名词优先于抽象治理词。**  
   观众更容易理解的是岗位、频道、项目空间、工牌、代码库、客户资料、日志、管理员，而不是身份模型、访问控制、治理体系。专业词可以出现，但必须先有现实对象托住。
5. **封面修改后必须同步发布包。**  
   改完封面大标题后，必须同步更新 `publish-copy.md`、短标题、QA 报告和视频号封面 JPG。否则发布时会出现封面、标题和简介口径不一致。
6. **抽帧检查要看逻辑是否换挡成功。**  
   本条抽帧必须能看出：不是 Agent -> AI 员工进入组织 -> 不借个人账号 -> 工牌机制 -> 六问检查。只检查有没有重叠还不够，还要检查主线是否已经从“功能”升级到“组织角色”。

本次沉淀出的通用判断：

- 如果用户说“这不是讲功能，而是讲背后的理念”，就不要再围绕按钮、入口、参数和单个机制组织视频。
- 如果视频里的核心词需要解释太久，例如“透传”“身份模型”“访问治理”，要改成角色变化、工作变化或现实流程变化。
- 新功能最好的观众收益通常不是“知道它发布了”，而是“知道自己以后设计类似系统时该先问什么”。

## 2026-06-30 Series Repair Lessons: EP04 And Later

The biggest lesson from later episode repairs is that production quality is not only about pretty frames. It is about keeping promise, visuals, subtitles, and final QA aligned.

1. **Lock the episode promise first.**  
   Before fixing a frame, check whether the cover, title, voiceover, and visuals all serve the same episode topic. A Memory episode must look like Memory; a cost episode must look like cost control.

2. **The cover must show the problem solved.**  
   A good cover is not just attractive. It tells viewers why this episode matters.

3. **Connect abstract components to real actions.**  
   Memory should look like handoff notes and state files. Automation should look like discovery, triage, and dispatch. Sub-agents should show maker and checker separation.

4. **Do not make subtitles carry the visual.**  
   If a diagram does not work without subtitles, the diagram is not ready. Subtitles should support, not rescue.

5. **Final QA must inspect final video frames.**  
   Source images can look fine while final subtitles, footers, player-safe areas, or encoding create new collisions.

6. **Clean stale segments before re-rendering.**  
   Old `s*.png` and `s*.mp4` files can corrupt QA. Regenerate contact sheets only from `segments.txt`.

7. **Prefer clean backgrounds over fake technical texture.**  
   Grid, checker, and noise textures usually reduce readability. Use solid backgrounds, clear cards, and focused highlights.

8. **English terms and arrows are core information.**  
   `Claude Code` becoming `ClaudeCode`, or an arrow losing its head, breaks trust and understanding.

9. **Publish copy follows the final cover.**  
   If the cover headline changes, update title, short title, description, QA notes, and platform cover exports together.
