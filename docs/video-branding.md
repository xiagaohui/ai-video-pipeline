# 短视频固定开场与结束语

后续短视频默认保留这两句固定话术。

## 最高优先级原则

这三条是所有视频制作的第一门禁，优先级高于选题形式、视觉风格、时长和工具实现。

1. **立意必须从观众视角出发，真的有价值，并且便于传播。**  
   立意不能只是“我想讲什么”或“材料里有什么”，而要回答“观众看完能得到什么”。价值可以是一个可实操的方法、一个新的判断框架、一个能透传的新观点，或者一个能解决真实痛点的具体建议。好立意还必须能被普通观众一句话转述，最好带有纠偏、反差或行动建议。没有观众收益、不能形成一句清楚主判断的选题，不进入口播和画面制作。

2. **表达必须从观众视角出发，让观众看得懂、听得懂。**  
   口播、标题、字幕、封面和配图都要避免过于抽象。能用现实工作和生活中的名词，就不要用空泛概念；能用具体角色、动作、场景、问题，就不要用内部术语。专业性可以保留，但必须翻译成观众能理解的语言。

3. **配图必须简洁，并且图内、图与图之间都有清楚逻辑。**  
   让观众能够看懂永远是第一优先级。每张图只服务一个核心意思，图内元素必须有明确阅读顺序、因果、对比或流程关系；前后图片必须沿同一条主线递进，不能各讲各的，也不能为了显得专业而堆过多模块和小字。

## 口播去 AI 味门禁

口播稿进入正式配音前，必须先做一次去 AI 味处理；如果重写了主线、标题或关键段落，必须再做一次。

默认 skill 选择：

| 场景 | 默认 skill | 用法 |
| --- | --- | --- |
| 中文短视频口播、标题、简介 | `writing-humanizer` | 主入口。保留立意和事实，清理模板句、讲义腔和过于工整的段落节奏。 |
| 已确认稿的保真修补 | `de-ai-prompt-enhancer` | 不改主张，不新增事实，只去掉路标词、二分对照壳、协作口吻和模型腔。 |
| 广谱 AI 痕迹审阅 | `humanizer-zh` | 当稿子读起来像新闻稿、百科或宣传文时，用它做问题扫描。 |
| 最后一遍节奏压缩 | `stop-slop` | 用来删空话、打破公式结构、检查句子是否太像提纲扩写。 |
| 多语言或批量文本 | `patina` | 重型编排器，适合批量审计或多语言稿件；短视频口播不作为默认入口。 |
| 整篇文章生产 | `writing-agent` | 用于完整写作工作流，不作为单纯去 AI 味工具。 |
| 人物视角/思维框架 | `nuwa-skill` | 用来生成观点视角，不用于常规去 AI 味。 |

去 AI 味不是把稿子改得松散，也不是加入虚假的个人经历。它只做四件事：

- 保留事实、观点和信息密度。
- 把讲义式句子改成自然口播。
- 把抽象词换成现实名词，例如需求、代码、测试、验收、日志、工具权限。
- 让每段都能回答“观众听完能得到什么”。

必须重点清理：

- 路标词：`接下来我们`、`下面来看`、`总的来说`、`值得注意的是`。
- 模板句：反复使用 `不是 A，而是 B`、`真正重要的是`、`本质上`、`核心在于`、`说白了`。
- 内部术语：没有解释的 `编排层`、`协同模式`、`上下文压缩`、`能力曲线`。
- 提纲腔：连续几段都是“观点句 + 解释 + 段尾总结”。
- 假亲近：无必要的 `你有没有发现`、`是不是很有启发`、`赶紧收藏`。

传播钩子可以有反差，但不能让整篇稿子都变成同一种句式。比如“多 Agent 不是越多越强”可以作为开头判断，后面必须落到具体判断标准和行动建议。

## 口播音频标准

后续知识类短视频默认采用“自然知识口播”标准：

- 语气：克制、清楚、可信，像一个懂行的人在解释，不做情绪化播报。
- 语速：适中偏紧凑，约每秒 5 个汉字；复杂概念前后要留短暂停顿。
- 句式：优先短句，一句只讲一个意思，避免长串抽象名词。
- 结构：观点先行，再解释含义，最后给观众可执行建议。
- 价值：每一节都回答“这对普通观众有什么用”，不能只复述新闻或访谈内容。
- 字幕对齐：口播、字幕、画面标题必须表达一致，避免同义改写导致观众听不懂。
- 发音校对：英文名、公司名、多音字、专有名词在生成音频前单独检查。
- 禁止：不要模仿或克隆未经授权的人声；可以参考公开视频的节奏和讲解方式。

## TTS 发音稿硬规则

生成 TTS 时可以维护一份“只给 TTS 读”的发音稿，但必须严格遵守：

- 发音稿只用于控制发音、停顿和连读，不得改变正式口播稿、字幕、画面标题和导出文案。
- 英文专有名词、人名、公司名、产品名必须保持英文，不翻译成中文，不拆成中文音译。例如 `Claude Code`、`GStack`、`Garry Tan`、`Anthropic`、`Dario Amodei` 必须保留英文写法。
- 缩写只能做最小发音辅助：可以去掉两侧多余空格，或展开为官方英文全称以便连读。例如 `YC` 可以在发音稿中写成 `Y Combinator`；不得写成中文译名。
- `AI` 默认保留英文缩写，并去掉两侧多余空格，避免 TTS 读出明显停顿。只有在用户明确要求时，才允许尝试其他发音写法。
- 免费 TTS 无法保证英文专名发音时，先生成 30-60 秒样音由主 agent 复听自检；只有换用付费引擎、克隆人声、或用户明确要求选择声音时，才需要停下来让用户确认。
- 发音稿和正式字幕可以不同，但含义必须一致；所有差异必须只服务发音，不得为了 TTS 顺口而改写事实、观点或术语。

## 固定开场

大家好，欢迎来到今天的 AI 工程观察。

## 固定结束语

以上就是今天的内容，希望对你有启发，我们下期见。

## 使用规则

- 开场放在正片第一句前，建议独立 3-5 秒。
- 结束语放在正文收束之后，建议独立 4-7 秒。
- 不加入点赞、关注、转发等引流话术。
- 语气保持克制、专业、价值导向。

## 内容结构规则

- 每条知识视频必须先明确：核心立意、观众价值、合规边界。
- 核心立意必须通过“一句话转述测试”：观众能用一句人话复述，标题、封面和开头都围绕同一句话展开。
- 传播性强的立意通常包含纠偏、反差或行动建议。表达时优先用观众能转述的人话，例如“多 Agent 要不要上？先看 4 个信号”“新功能解决的不是炫技，而是团队提效”“先别急着搭系统，先把验收标准写清楚”。
- 每个重点观点默认采用“观点 -> 意味着什么 -> 给观众的建议”的结构。
- 口播不是复述材料，而是把复杂材料翻译成观众能带走的判断。
- 新功能类视频必须先讲“解决什么问题”，再讲“功能是什么”。例如官方新功能不能只说“发布了 Artifacts”，要说它如何带来团队提效。
- AI 员工类新功能不能停留在“做了一个 Agent”或“多了一个按钮”。例如 Claude Tag 的重点不是 `@Claude` 或单个 Agent 功能，而是 AI 开始作为组织里的员工进入真实工作环境；视频必须讲清岗位、工位、工牌、工具、记录和接管机制。`Agent identity` 只是支撑这个变化的工牌机制，不是最终立意。
- 理念型新功能要先讲“角色变化”或“工作方式变化”，再讲具体机制。合格表达是 `AI 从个人助手变成组织员工`；不合格表达是只围绕 `Agent identity`、`权限模型`、`Tag 功能` 做机制说明。
- 热点概念类视频必须先写清“常见误区”和“落地方法”。例如 Loop Engineering 不是让 AI 多跑几轮，而是让 loop 可恢复、可观测、可接管。
- 避免内部提纲式表达。标题、封面标签和字幕要尽量使用观众能直接理解的问题或判断。
- 如果需要拆分“角色”或“团队分工”，优先按真实业务或研发责任链设计，例如“产品负责人、技术负责人、设计负责人、开发工程师、测试工程师、发布负责人”。避免“反方评审、工作流成员”这类内部术语；风险审查应放进技术负责人或测试工程师的职责里，而不是硬造一个怪角色。

## 自动自确认门禁

后续视频制作默认由主 agent 按规范自审、自改、自确认，不再把口播稿、分镜、封面、抽帧或发布包作为常规中间产物提交给用户等待确认。用户给出选题、链接或立意后，主 agent 必须一路推进到可交付成片、封面、QA 和发布文案。

只有以下情况需要暂停询问用户：

- 需要用户登录、扫码、授权、输入账号密码、支付或开通付费服务。
- 需要使用或克隆用户本人声音、肖像、隐私素材，且授权边界不明确。
- 来源事实无法确认，继续制作会有明显误导风险。
- 涉及法律、医疗、金融、国家安全、平台红线等高风险内容，需要用户明确选择表达边界。
- 用户明确要求“先给我看”“等我确认”“不要继续”。

## 口播稿自审门禁

- 口播稿写完后，必须先通过自审门禁，不得直接进入配音、字幕或视频合成。
- 必须启动多个子 agent 或等价独立审阅视角，从观众视角审阅口播稿，并汇总修改意见。
- 子 agent 至少覆盖三种视角：普通观众是否听得懂、目标受众是否觉得有价值、平台发布是否存在误读或合规风险。
- 审稿输出必须包括：听不懂的句子、价值不足的段落、标题/字幕/口播不一致之处、建议改法。
- 主 agent 必须吸收审稿意见完成修订，并记录“自确认通过”的理由。
- 自确认通过后，直接进入正式配音、字幕和视频合成，不再等待用户确认。

## 硬规则应用门禁

硬规则按“节点门禁”应用，不只在最后总检时应用。每个可交付节点完成后，必须先通过对应检查，再进入下一步。

| 节点 | 产物 | 应用的硬规则 | 审阅方式 | 是否阻塞下一步 |
| --- | --- | --- | --- | --- |
| 立意与结构 | 立意、观众价值、合规边界 | 必须从观众视角出发；必须能说明可实操方法、新观点或真实痛点收益；必须能被一句话转述；不能只复述材料；每节必须有观众收益；敏感内容先定边界 | 主 agent 自检；必要时从普通观众视角独立审阅；敏感/高风险内容加独立合规视角 | 是 |
| 口播稿 | 正式口播稿 | 短句、自然、观点->含义->建议；标题/字幕/画面同义；英文专名保留英文 | 必须启动多个子 agent 或等价独立视角，从普通观众、目标受众、平台风险审阅；主 agent 修订并自确认 | 是 |
| 画面脚本 | 分镜、每屏标题、配图说明 | 每屏服务章节；不使用抽象装饰图；画面标题和口播一致；配图简洁、图内有顺序、前后图有递进 | 主 agent 自检；复杂信息图或新视觉风格加独立视觉审阅 | 是 |
| 发音稿与样音 | TTS 发音稿、30-60 秒样音 | 发音稿只服务发音；英文专名不翻译不音译；多音字和缩写单独检查 | 主 agent 校对并复听样音；换付费引擎、人声克隆或授权不清时才询问用户 | 是 |
| 正式音频 | 完整音轨 | 自然度、停顿、英文连读、响度、时长匹配 | 工具检查时长/响度；主 agent 复听重点片段 | 是 |
| 视频初版 | MP4、抽帧图、截图 | 9:16 无黑边；标题/图/字幕不重叠；底部安全区不放关键文字 | 工具检查参数；独立视觉 QA 或截图审阅检查重叠与安全区 | 是 |
| 封面 | 封面 PNG/SVG | 一图概括内容；标签用观众收益表达；无重叠、贴边、裁切 | 必须肉眼检查；信息图封面加独立视觉审阅 | 是 |
| 发布包 | 视频、封面、标题、简介 | 标题/封面/视频承诺一致；没有违规素材和未经授权人声 | 最终清单自检；敏感内容加合规复核 | 是 |
| 交付归档 | 主题文件夹 | 成片、封面、QA、口播、字幕、音频和源工程放在同一主题目录；不复制 node_modules | 清单检查；生成 MANIFEST | 是 |

独立 agent 用在“主观判断容易漏”的节点：口播是否听得懂、画面是否真的清楚、封面是否能一眼看懂、平台表达是否有误读风险。纯机械节点，例如分辨率、时长、编码、黑边初筛，优先用工具检查。

## 画面与封面规则

- 重点章节配图必须服务章节内容，优先使用信息图、对比图、流程图、清单图，不使用空泛科技感背景。
- 配图第一优先级是观众能看懂。图内必须有阅读顺序，前后图必须递进；宁可少放内容，也不要让观众猜图在讲什么。
- 每张图做 3 秒扫视测试：能不能看出主题、主线和观众收益。看不出来就重画。
- 模式、架构、框架类画面必须“结构先行，文字后置”。每页优先画出架构关系、数据流、分工流或决策流，再用短标签补充适用场景、优势和风险。不要把每种模式做成一堆文字卡片。
- 模式讲解页默认采用固定结构：上方一句人话标题，中间一张架构图，下方三枚短标签 `适用 / 优势 / 风险`，再补一个现实场景。三枚短标签每枚只写一个短句，不能写段落。
- 过程图的主视觉必须能独立表达“这个模式怎么运转”。例如生成验证模式要看见 `生成 -> 验证 -> 通过/重写`，子 Agent 模式要看见 `主 Agent -> 多个专项 Agent -> 统一结论`，总线模式要看见 `事件 -> Bus -> 对应 Agent`。
- 如果观众看图仍需要先读完一堆小字才知道模式是什么，说明图不合格，必须重画架构，而不是继续改文案。
- 9:16 竖屏视频必须检查手机观看效果：标题、图片、字幕不能重叠，底部关键文字不能被播放器控制条遮挡。
- 导出后必须检查右侧、底部、上下边缘是否有黑边。
- 封面必须包含主题来源、主判断、观众收益点和内容证据；封面主标题要服务同一句可传播立意，底部标签优先使用问题式表达。
- 封面主标题默认采用“权威来源给出的方法/案例 + 这个方法解决的观众痛点”的结构。大厂背书不能只是贴品牌名，必须和痛点有因果关系：Google 给出的流程解决上线失控，Claude 给出的模式解决选择困难，Anthropic 的实战解决代码并行混乱，Claude Code 官方方法解决 Subagent 入门和权限失控。
- 封面最好形成明确反差：权威团队或官方案例把同类任务做得很稳，但观众自己的 AI / Agent 总出问题。标题必须写出同一个原因，例如“Anthropic 让 16 个 Claude 写出 10 万行代码；你的 AI 写乱，是因为缺测试和隔离”。
- 反差标题必须有因果桥，不能把“大厂案例”和“观众痛点”硬拼在一起。合格句式是“权威案例靠 A 做成 B；你的问题是缺 A”。例如“Anthropic 让 16 个 Claude 写出 10 万行代码；你的 AI 写乱，是缺测试和隔离”。
- 痛点必须是观众自己的问题，而不是案例本身的结果。优先写“上线总失控”“写代码总改乱”“不知道先建哪个”“调研结果全是废话”这类可感知麻烦；少写“为什么没有翻车”“到底强在哪里”这类旁观式问题。
- 封面大标题必须优先使用观众能直接感知的痛点、反差或问题，例如“越做越乱”“上线翻车”“为什么乱改”“一堆废话”。避免使用需要解释的内部词或概念词，例如“任务形状”“协同模式”“编排层”“上下文压缩”。
- 流程类或方法论类视频，封面优先做成“一图概括整条视频”的信息图，但必须控制信息密度。
- 新功能类封面必须同时体现“最新功能”和“它解决的问题”，不要只写功能名。
- 理念型新功能封面可以采用 `产品名核心理念：旧角色/旧方式变成新角色/新方式`。这类标题必须直接描述观众能理解的变化，例如 `Claude Tag核心理念：AI从个人助手变成组织员工`，避免只写 `透传`、`底层逻辑`、`身份模型` 这类需要二次解释的词。
- 架构类或热点概念类封面必须突出立意，优先采用“误区 -> 正确结构 -> 落地收益”的表达，不只罗列组件。
- 封面导出后必须肉眼检查：主标题、信息图节点、底部总结条、标签、角标之间不能互相覆盖、贴边或被裁切。
- 封面主标题变化后，必须同步更新视频号标题、短标题、简介、QA 报告和封面 JPG；不能只替换 PNG，导致发布包口径不一致。
- 竖屏封面必须使用完整画幅，不能把主体内容集中在上半屏、下半屏只留背景空白。底部区域必须承担明确功能：结论、观众收益、行动建议或关键证据。
- 可有可无的封面元素，例如时长、装饰角标、重复标签，只要影响核心信息阅读就删除。
- 封面标签必须使用观众能理解的收益表达，例如“需求不盲做”“风险先评审”“真实页面验收”“发布有门禁”，避免只写内部章节名。

## 发布文案规则

- 视频号短标题必须优先短、狠、能懂。默认控制在 12 个汉字以内；平台或用户要求 16 个字符以内时，英文、数字、空格和标点都算字符。
- 短标题不要把完整封面标题照搬进去。封面可以承担主钩子，短标题只保留最能被搜索和转述的核心词。
- 短标题不够短时，优先删来源修饰词、形容词和重复词，保留对象和问题。例如 `Claude 官方给了 5 种 Agent 模式` 可压成 `5种Agent模式` 或 `Agent模式怎么选`。
- 视频描述第一句必须承接封面承诺，先说观众能得到什么，再说明来源和内容结构。
- 话题标签优先 6-10 个，覆盖主题、工具、受众和平台热词；不要堆无关泛标签。

## 深色高冲击视觉模板

用户给出的参考图说明：深色科技风可以使用，但必须是“清楚的信息图”，不是抽象背景。

默认模板：

- 背景：深蓝/近黑渐变，允许微弱网格、代码框、流程线、盾牌、窗口轮廓等低透明背景元素。
- 主标题：白色超大字，1-2 行，必须是观众能直接理解的问题或动作判断。
- 左侧锚点：可用一条彩色竖线 + 小标签标明本页功能，例如 `为什么组合`、`一个原则`、`用 GitHub 的话`。
- 主视觉：只放一个核心结构，例如双卡对比、四步列表、左右风险清单、三段流程。
- 当前重点：用发光边框、高亮底色或加粗色块标出，不要只靠口播说明。
- 颜色语义：
  - 蓝色：主工具、主流程、执行能力。
  - 青色：审查、查问题、信息过滤。
  - 紫色：第二视角、复审、判断。
  - 橙色：风险、警告、边界。
  - 绿色：通过、推荐、正确做法。
- 字幕与平台遮挡：底部最多放强化句，核心信息必须在中上部完成表达。

不使用：

- 每一页都做三张同款卡片，缺少页面功能差异。
- 小字过多、解释塞满卡片。
- 只有发光和图标，没有真实动作或判断。
- 把封面和过程图都做成同一种排版，导致节奏单调。

后续制作时，封面可以更“冲击”，过程图要更“讲清楚”。封面优先大结论、大数字、大反差；过程图优先对比、步骤、风险清单和高亮当前步骤。

## Post-EP04 Visual Repair Rules

These checks were added after repeated real production fixes. Apply them to every future video:

- Series episodes must match the planned episode promise. Cover, title, voiceover, visuals, and publish copy cannot drift.
- Every process frame needs a visible subject and action, not just abstract component names.
- Keep the visual body, takeaway bar, and subtitle safe area separated. Do not place core nodes, arrows, or conclusions in the subtitle area.
- Run collision checks: titles, cards, nodes, arrows, lines, and takeaway bars need safe spacing; arrowheads must be visible.
- Use solid or clean dark backgrounds by default. Do not rely on grid, checker, or noise textures for technical style.
- Before every re-render, clear stale segments. QA should inspect actual `segments.txt` clips and final MP4 frames.
- Subtitles are max two lines, not clipped, and must not cover the image. Fix layout before shrinking subtitle text.
- Scene switches must follow final audio/VTT timing, not storyboard estimates.
- Preserve English term spacing: no `ClaudeCode`, `ContextEngineering`, or `PromptEngineering`.
- When the cover headline changes, update publish title, short title, description, and platform JPG exports.

## 交付归档规则

- 每条视频完成后，按主题创建独立交付文件夹。
- 文件夹默认包含：`final/` 成片和无音频视觉版，`cover/` 封面 PNG/SVG，`qa/` 抽帧或截图，`build/` 口播、字幕、音频中间文件，`source/` 源工程或生成脚本。
- 保留原始工作目录，只复制交付包，避免打断已有路径引用。
- 不复制 `node_modules`、缓存、临时逐帧文件等可再生成的大体积依赖。
- 每个交付总目录生成 `MANIFEST.txt` 或等价清单，方便后续发布、复盘和重剪。

## 合规与素材规则

- 不直接搬运未经授权的原片画面或克隆未经授权的人声。
- 涉及医疗、金融、法律、安全、岗位变化等话题时，只给风险边界和判断建议，不做确定性承诺。
- 海外访谈类内容要避免把冲突、军事、地缘政治做成卖点；保留可公开讨论的工程判断和工作建议。
